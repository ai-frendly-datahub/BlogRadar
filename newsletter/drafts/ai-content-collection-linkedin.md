# AI 콘텐츠 수집 글 초안 메모

이 문서는 BlogRadar 기반 발행물 외에, 소셜 포스트나 링크드인용 글감으로 재가공할 수 있는 수동 초안을 모아두는 위치다. 아래 내용은 "AI 콘텐츠 수집 도구 조합" 글감을 링크드인/소셜 게시글 관점으로 다듬은 초안이다.

## 편집 방향 메모

이 글은 소재가 좋고, 실제 사용감이 살아 있어서 관심은 끌 수 있는 초안입니다.
다만 지금 상태로 올리면 읽는 사람 입장에서는 두 가지가 걸릴 수 있습니다.

첫째, 도구 설명은 많은데 메시지 중심축이 약합니다.
둘째, 과한 표현 몇 군데가 신뢰를 깎습니다.
예를 들어 `10분이면 끝`, `1인으로도 모든 회사 업무를 다할 수 있는 환경`은 인상은 강하지만 반감도 함께 부를 수 있습니다.

그래서 방향은 간단합니다.
도구 소개 글이 아니라 `내 업무 흐름이 어떻게 바뀌었는가`로 재구성하면 훨씬 강해집니다.

## 먼저 손봐야 할 포인트 4개

### 1) 제목은 더 날카롭게

지금도 나쁘지 않지만, 왜 지금 깔아야 하는지가 더 바로 보여야 합니다.

추천 제목:

- `AI 콘텐츠 수집, 아직도 손으로 모으시나요? 제가 먼저 깔아두는 5가지`
- `콘텐츠 리서치 속도 3배 올려준 도구 5개`
- `AI 에이전트 시대, 콘텐츠 수집은 이 조합으로 끝냅니다`

### 2) 도구 나열보다 `문제 → 해결` 구조로

사람들은 도구 이름보다 내가 겪는 불편이 해결되는지를 먼저 봅니다.
그래서 각 항목은 이 구조가 좋습니다.

- 무엇이 불편했는지
- 이 도구가 뭘 해결하는지
- 실제로 어디에 쓰는지

### 3) 과장 표현은 성과형 문장으로 교체

예:

- `10분이면 끝납니다` → `초안 제작 시간이 눈에 띄게 줄었습니다`
- `모든 회사 업무를 다할 수 있는 환경` → `리서치와 정리 업무의 상당 부분을 1인 체제로 돌릴 수 있게 됐습니다`

이렇게 바꾸면 힘은 유지하면서 신뢰가 올라갑니다.

### 4) 마지막 문장은 자기 자랑보다 독자 행동 유도

지금 마무리는 기대감은 있지만, 댓글이나 저장을 유도하는 힘은 약합니다.

예:

- `원하시면 다음 글에서 제가 실제로 쓰는 수집 → 요약 → 보고서 제작 흐름까지 공개해보겠습니다.`
- `반응 좋으면 제가 쓰는 프롬프트와 자동화 흐름도 정리해보겠습니다.`

## 바로 올릴 수 있게 다듬은 버전

### 버전 A. 가장 추천하는 형태

조금 더 날카롭고, 링크드인에서 반응 나오기 좋은 톤입니다.

```text
🔍 AI 콘텐츠 수집, 아직도 수동으로 하고 계시면 이 5개부터 보셔야 합니다

요즘 “콘텐츠는 어떻게 모으고 관리하세요?”라는 질문을 자주 받습니다.

저는 요즘 아래 5가지를 중심으로 씁니다.
핵심은 하나입니다.

“수집”에서 끝나는 도구가 아니라,
수집한 데이터를 AI 에이전트가 바로 읽고 처리할 수 있는 형태로 바꿔주는가.

이 기준으로 보면, 아래 조합이 꽤 강력합니다.

1️⃣ Agent-Reach
소셜 미디어 데이터를 한곳으로 모아주는 멀티 플랫폼 수집기

X, LinkedIn, Instagram 등 여러 채널의 포스트와 반응 데이터를 한 인터페이스에서 모을 수 있습니다.
플랫폼별로 따로 스크래퍼를 붙이던 방식보다 훨씬 운영이 단순해집니다.

제가 가장 자주 쓰는 용도는 경쟁사 콘텐츠 모니터링입니다.

• 여러 소셜 채널 데이터를 한 번에 수집
• 반응 데이터까지 함께 추적 가능
• 트렌드 조사, 경쟁사 분석에 바로 투입 가능

2️⃣ web-access
Claude Code에 웹 브라우징 능력을 붙여주는 Skill

Claude Code를 쓰다 보면 인터넷 접근이 막히는 순간 답답할 때가 많은데,
이 Skill은 그 지점을 꽤 잘 메워줍니다.

CDP 기반 브라우저 조작이 가능하고,
사이트별 접근 경험이 누적되는 구조라 반복 작업일수록 효율이 올라갑니다.

• Claude Code에 실시간 웹 접근 능력 추가
• 반복 접속 작업에 유리
• 조사형 워크플로우와 궁합이 좋음

비슷한 방향으로는 bb-browser도 흥미롭습니다.
이쪽은 브라우저 자체를 API처럼 다루는 접근입니다.

3️⃣ r.jina.ai
웹페이지를 깔끔한 Markdown으로 바꿔주는 변환기

웹페이지를 LLM에 넣으려면 HTML 정리가 늘 번거롭습니다.
r.jina.ai는 이 과정을 거의 없애줍니다.

URL 앞에 붙이기만 하면 광고, 내비게이션 같은 노이즈를 덜어낸 Markdown 형태로 바꿔주기 때문에
Claude Code나 Codex에 바로 넣기 좋습니다.

• 웹페이지를 Markdown으로 정리
• LLM 컨텍스트 투입에 최적화
• 별도 파싱 코드 부담 감소

4️⃣ last30days-skill
최근 30일 콘텐츠를 자동으로 훑고 정리해주는 Skill

개별 페이지 수집은 해결돼도,
“최근 한 달간 이 분야에서 무슨 일이 있었지?”를 한 번에 정리하는 건 또 다른 문제입니다.

이 Skill은 그 시간 범위 자체를 기준으로 콘텐츠를 훑고 요약해줘서
주간 트렌드 리포트나 뉴스레터 초안 만들 때 특히 유용합니다.

제가 가장 자주 쓰는 순간은 월요일 아침입니다.
“지난주에 뭐 놓쳤지?”를 빠르게 확인하기 좋습니다.

• 최근 30일 콘텐츠 스캔 및 요약
• 트렌드 리포트 초안 생성에 유리
• 놓친 이슈를 빠르게 복기 가능

5️⃣ 결국 중요한 건 ‘AI가 바로 읽을 수 있는 형태’입니다

Agent-Reach는 소셜 데이터를,
web-access는 실시간 웹 정보를,
r.jina.ai는 정적 페이지를,
last30days-skill은 시간 범위 기반 콘텐츠를

각각 AI 에이전트가 바로 처리할 수 있는 형태로 바꿔줍니다.

저는 지금
맥북에서 Codex나 Claude Code로 즉시 조사하는 방식과,
원격 서버에 에이전트를 띄워두고 메신저로 조사 요청을 던지는 방식을 함께 씁니다.

덕분에 리서치 → 정리 → 초안 제작까지 걸리는 시간이 크게 줄었습니다.

다만, 한 가지는 분명합니다.
수집과 정리가 아무리 빨라져도 결국 내용을 직접 읽고 이해하지 않으면
겉모양만 번지르르한 결과물만 남습니다.

속도가 생기면, 그다음은 해석력입니다.

좋은 시그널이 보이는 글은 꼭 정독해야 합니다.

반응 좋으면 다음 글에서
제가 실제로 쓰는 “수집 → 요약 → 보고서/PPT 초안화” 흐름도 공개해보겠습니다.
```

### 더 짧고 강한 버전

조금 더 바이럴 지향입니다.

```text
🔍 AI 콘텐츠 수집, 아직도 손으로 모으시나요?

저는 요즘 이 4개+1원칙으로 정리합니다.

1. Agent-Reach
→ 소셜 데이터를 여러 플랫폼에서 한 번에 수집

2. web-access
→ Claude Code에 웹 브라우징 능력 추가

3. r.jina.ai
→ 웹페이지를 LLM 넣기 좋은 Markdown으로 변환

4. last30days-skill
→ 최근 30일 콘텐츠를 시간 범위로 스캔하고 요약

5. 공통 원칙
→ 중요한 건 “많이 모으는 것”이 아니라
   “AI 에이전트가 바로 읽고 처리할 수 있는 형태로 바꾸는 것”

저는 지금
로컬에서는 Codex/Claude Code로 바로 조사하고,
원격에서는 에이전트를 띄워 메신저로 리서치를 돌립니다.

이 조합의 장점은 명확합니다.
리서치 → 정리 → 초안화까지의 마찰이 크게 줄어듭니다.

다만 속도와 이해도는 별개입니다.
수집이 빨라졌다고 읽기를 건너뛰면 결과물은 얕아집니다.

좋은 시그널이 보이는 글은 결국 직접 읽어야 합니다.

반응 좋으면 다음 글에서
제가 실제로 쓰는 자동화 흐름도 공개하겠습니다.
```

## 이 글이 더 강해지려면 꼭 추가하면 좋은 것

딱 3개만 붙이면 완성도가 확 올라갑니다.

1. 실제 전후 비교 한 줄

- `예전엔 플랫폼별로 따로 모았는데, 지금은 한 흐름에서 정리합니다.`
- `예전엔 링크만 쌓였고, 지금은 바로 요약 가능한 형태로 들어옵니다.`

2. 실사용 예시 한 줄

- `경쟁사 주간 콘텐츠 모니터링`
- `월요일 아침 업계 동향 복기`
- `뉴스레터 초안 재료 수집`

3. 콜투액션 한 줄

- `원하시면 제가 쓰는 실제 워크플로우도 다음 글에서 정리하겠습니다.`
- `댓글 주시면 실사용 흐름 예시도 이어서 올리겠습니다.`

## 한 줄 총평

지금 초안은 도구를 많이 아는 사람의 글처럼 보입니다.
조금만 손보면 실제로 업무를 바꿔본 사람의 글로 바뀝니다.

핵심은 이겁니다.
도구 소개를 줄이고, 업무 변화의 증거를 늘리세요.
